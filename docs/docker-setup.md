# Headless Cocos Preview — Docker Setup

Run the headless Creator 3.8 preview stack (`preview-mirror.mjs` + mini-packer +
engine snapshot) inside a Linux container. Verified against the
`baseAIAutoCocos` sample project on **Docker Desktop 4.71 (Windows)** with
the Linux engine — should work identically on native Linux Docker and on
Docker Desktop for macOS.

## What runs inside the container

```
/app
  preview-mirror.mjs          HTTP + WebSocket (port 7460)
  packer/build.cjs            mini-packer, spawned by mirror per rebuild
  cache/                      preview shell (index.html, settings.js, …)
  node_modules/               trimmed Cocos-programming deps (~21 MB)
  vendor-utils/dist/uuid.js   compressUUID helper (Creator's, not on npm)
/engine/preview/              bundled cc engine (91 MB, one-time snapshot)
/D:/tempWorkspace/baseAIAutoCocos → /workspace   (symlink, see below)
/workspace                    Cocos project (bind-mount from host)
```

Total image size ≈ 1.7 GB (1.2 GB of that is `node:20-bookworm-slim`;
our own code + engine + npm tree = ~550 MB).

## Quick start

```powershell
cd d:\tempWorkspace\headless-cocos-research\docker
$env:DOCKER_BUILDKIT = "0"     # classic builder uses local base cache first
docker build -t cocos-headless-preview:latest .\build-context
docker compose up -d
# browser (or e2e script) → http://localhost:7460/
```

Container should reach `healthy` within ~15 s and log:

```
[preview-mirror] http://127.0.0.1:7460
[mini] mode=on script=/app/packer/build.cjs
[hmr] watching library /workspace/library (+poll)
[mini] watching /workspace/assets (+poll)
[asset-sync] boot: N written, M unchanged
[mini] build#1 ok in ~500 ms
```

## Verification

Three checks in ascending confidence:

```powershell
# 1. Mirror alive
Invoke-WebRequest http://127.0.0.1:7460/__hmr/status | % Content
# → {"clients":0,"seq":1,"watch":true}

# 2. Overlay serves the mini-packer's chunk
$body = (Invoke-WebRequest http://127.0.0.1:7460/scripting/x/chunks/7d/7d0661b2ee33e4b885a478dbc38c7441b8df4bbf.js).Content
$body -match 'PROBE_VERSION = (\d+)'   # → matches whatever is currently in HeadlessProbe.ts

# 3. Full HMR loop end-to-end
# edit assets/scripts/HeadlessProbe.ts on the Windows host → wait ~2 s → chunk changes
Set-Content D:\tempWorkspace\baseAIAutoCocos\assets\scripts\HeadlessProbe.ts ((Get-Content ... -Raw) -replace 'PROBE_VERSION\s*=\s*\d+','PROBE_VERSION = 42') -NoNewline
Start-Sleep 4
docker exec cocos-preview grep -o 'PROBE_VERSION = [0-9]*' /workspace/temp/programming/packer-driver/targets/preview-mini/chunks/7d/7d0661b2ee33e4b885a478dbc38c7441b8df4bbf.js
# → PROBE_VERSION = 42
docker logs cocos-preview | Select-String 'build#'
# → [mini] build#N start (scripts/HeadlessProbe.ts)
# → [mini] build#N ok in ~2500 ms
# → [hmr] browser:reload mini:scripts/HeadlessProbe.ts
```

## Design decisions worth remembering

### 1. Vendored `@cocos/creator-programming-*` deps

None of the mini-packer's core dependencies (`@cocos/creator-programming-quick-pack`,
`…mod-lo`, `…common`, `…import-maps`, `…babel-preset-cc`,
`@cocos/creator-programming-babel-plugin-dynamic-import-vars`) are on npm.
The image ships a **traced** subset of `tmp-asar-root/node_modules` (180
packages, 21 MB) that Node actually loaded during a real build.

To refresh:

```powershell
cd d:\tempWorkspace\headless-cocos-research\spike\packer
node _trace-deps.cjs                    # writes docker/loaded-files.json
# staging is (re)generated by hand in the docker folder; see build-context/vendor/
```

`ws` is not part of the Cocos tree — it's added on top so the mirror can
serve WebSocket HMR.

### 2. Windows-style chunk hashes (`PROJECT_URL`)

Cocos Creator computes chunk filenames as `sha1(<module URL>)`. Creator on the
Windows host produces URLs like `file:///D:/tempWorkspace/baseAIAutoCocos/…`;
`pathToFileURL('/workspace/…')` in the container would produce different URLs
and therefore different chunk hashes. Two things keep everything aligned:

- **`PROJECT_URL=file:///D:/tempWorkspace/baseAIAutoCocos`**
  Passed to `build.cjs`; it substitutes the URL prefix so mini-packer's
  chunks land at the same hash Creator's `preview/import-map.json` already
  points to. See `moduleUrlFor()` in `spike/packer/build.cjs`.

- **Container symlink `/D:/tempWorkspace/baseAIAutoCocos → /workspace`**
  mod-lo `stat()`s file:// URLs during resolution and doesn't consult its
  own load mappings for that step. On Linux, `fileURLToPath('file:///D:/…')`
  yields `/D:/…`. The Dockerfile creates the symlink at build time; it goes
  live as soon as the bind-mount populates `/workspace` at run time. The
  packer also passes `preserveSymlinks: true` to mod-lo so URLs don't get
  rewritten back to `/workspace/…` form.

Both are only needed when the container is meant to co-exist with a
Creator-generated `preview/import-map.json`. For a from-scratch Linux
project with no prior Creator run, drop `PROJECT_URL`; mini-packer will
produce a fresh, self-consistent import-map on its own.

### 3. `WATCH_POLL=1` for bind-mount FS events

Docker Desktop for Windows/macOS runs the container in a lightweight VM;
inotify events from host edits **do not** cross into it. The mirror ships
with a directory-mtime-based polling watcher that runs alongside native
`fs.watch`:

- `WATCH_POLL=1`   enable polling in addition to native watch
- `WATCH_POLL_MS=800` scan interval (ms)

On native Linux Docker, either drop `WATCH_POLL` or leave it — the extra
scan is cheap on ~1000 files.

Because Docker Desktop's mtime propagation can jitter, an edit sometimes
triggers **two** builds (the second one lands as `build#N pending` in
logs). Harmless; both builds produce the same chunk.

### 4. Engine snapshot vs live Creator cache

`/engine/` is a one-time copy of Creator's warmed preview cache
(`C:\ProgramData\cocos\editors\Creator\3.8.8\resources\resources\3d\engine`).
Baked into the image so the container is self-sufficient. **If Creator
upgrades minor version, rebuild the snapshot and rebuild the image.**

```powershell
Remove-Item -Recurse d:\tempWorkspace\headless-cocos-research\spike\engine-snapshot
mkdir d:\tempWorkspace\headless-cocos-research\spike\engine-snapshot
Copy-Item -Recurse C:\ProgramData\cocos\editors\Creator\3.8.8\resources\resources\3d\engine\* d:\tempWorkspace\headless-cocos-research\spike\engine-snapshot\
# then re-copy into build-context/engine and docker build again
```

## Measured behavior

| Metric | Native (Windows host) | Docker Desktop (WSL2) |
| ------ | --------------------- | --------------------- |
| Boot to `mirror ready`    | ~0.5 s | ~2 s |
| First mini-build (`build#1`) | ~500 ms | ~2–4 s |
| Steady-state incremental build | ~250 ms | ~2–2.5 s |
| File-edit → chunk on disk | ~500 ms (native watch) | ~1.5 s (poll) + build |
| Idle image RAM | n/a | ~120 MB |

The Docker Desktop slowdown is entirely the bind mount / gRPC-FUSE layer.
Native Linux Docker should perform close to bare metal.

## Known limitations

1. **`temp/` writes leak to the host.** The container writes
   `temp/programming/packer-driver/targets/preview-mini/**` under the bound
   project. Fine for dev, but if you run the container in CI where the
   project is read-only, mount an additional writable tmpfs at
   `/workspace/temp` (or a named volume) to isolate.

2. **Chromium / puppeteer not in the image.** E2E tests (`e2e-hmr.cjs`,
   `e2e-scene-edit.cjs`) currently run from the Windows host against the
   container-served preview. Add `RUN apt-get install -y chromium` and
   set `PUPPETEER_EXECUTABLE_PATH=/usr/bin/chromium` if you want fully
   contained testing.

3. **Windows drive-letter symlink is single-project.** The Dockerfile hard-codes
   `/D:/tempWorkspace/baseAIAutoCocos → /workspace`. If you want to run other
   projects, either bake a different symlink target or run without
   `PROJECT_URL` (see §2 above).

4. **`preview-app/main.js` reload hook can double-fire under Docker's mtime
   jitter.** Debounce is `RELOAD_DEBOUNCE_MS=400`; bump if you see it in
   your workload.
